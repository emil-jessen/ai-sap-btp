# my-btp-app — Project Context

## What This App Is
A full-stack SAP BTP application with:
- **CAP backend** (Node.js) serving an OData v4 CatalogService
- **SAP UI5 Fiori frontend** (List + Detail views)
- **SAP Approuter** for authentication and routing
- Deployed to **Cloud Foundry** on BTP Trial (org: `cd3f5509trial`, space: `dev`, region: `us10-001`)

---

## Architecture

```
Browser
  └─> Approuter (my-btp-app-approuter)   ← handles XSUAA auth + routing
        ├─> HTML5 Apps Repo Runtime       ← serves UI5 frontend static files
        └─> CAP Backend (my-btp-app-srv)  ← OData /catalog/ endpoints
```

### Key service instances on BTP:
| Resource name | Service | Plan |
|---|---|---|
| `my-btp-app-xsuaa` | xsuaa | application |
| `my-btp-app-destination` | destination | lite |
| `my-btp-app-html5-host` | html5-apps-repo | app-host |
| `my-btp-app-html5-runtime` | html5-apps-repo | app-runtime |

## Important Naming (critical — must stay consistent)

| Setting | Value |
|---|---|
| `sap.app.id` (manifest.json) | `mybtp.ui` |
| `sap.cloud.service` (manifest.json) | `mybtp` |
| Approuter `welcomeFile` | `/mybtp.ui/index.html` |
| `sap.cloud.service` in mta.yaml destinations | `mybtp` |
| UI5 namespace (index.html, Component.js, controllers, views) | `mybtp.ui` |
| CAP service path | `/catalog/` |

> ⚠️ **Critical routing rule**: The SAP Approuter's dynamic routing extracts ONLY the part BEFORE THE FIRST DOT of the first URL path segment as the "service tag".
> - URL `/mybtp.ui/index.html` → service tag = `mybtp` ✅ matches `sap.cloud.service: mybtp`
> - URL `/my.btp.app.ui/index.html` → service tag = `my` ❌ "Service Tag my is unknown" → 503
>
> Therefore: `sap.cloud.service` must equal the text before the first dot in `sap.app.id`.
> With `sap.app.id = "mybtp.ui"` and `sap.cloud.service = "mybtp"` — routing works correctly.
>
> The html5-apps-repo-rt serves the app at `/{sap.app.id}/` (full ID as a single URL path segment, dots preserved).

## Deployment Steps

From the project root on **Windows** (requires CF CLI + mbt installed):

```bash
# 1. Build CAP backend
npm run build

# 2. Build UI5 frontend + create zip (already done — dist.zip is in resources/)
#    Only redo this if you change frontend source files:
npm run build --prefix app/ui
npm run zip --prefix app/ui
copy app\ui\dist\dist.zip resources\dist.zip

# 3. Build MTA archive
mbt build

# 4. Deploy to CF
cf login   # if not already logged in
cf deploy mta_archives/my-btp-app_1.0.0.mtar
```

## Known Issues Fixed

| # | Problem | Root Cause | Fix Applied |
|---|---|---|---|
| 1 | 503 "Service Tag mybtp is unknown" | `sap.cloud.service="mybtp"` but `sap.app.id="my.btp.app.ui"` — html5-rt redirected to `/my.btp.app.ui/index.html`, approuter extracted `my` as service tag | First attempt: changed service to `my.btp` and welcomeFile — still failed |
| 2 | 503 "Service Tag my is unknown" (after first fix attempt) | URL `/my.btp/app.ui/index.html` → approuter still extracts `my` (pre-dot of `my.btp`). Also html5-rt doesn't support multi-segment app IDs — it uses `/{sap.app.id}/` as a single path segment | Renamed UI5 namespace + sap.app.id from `my.btp.app.ui` → `mybtp.ui`, kept `sap.cloud.service=mybtp`. Now URL is `/mybtp.ui/index.html`, approuter extracts `mybtp` ✅ |
| 3 | 503 "Service Tag mybtp is unknown" (after namespace rename) | Destinations were at `instance` level in the Destination service, but the approuter's dynamic routing queries at `subaccount` level. Also `existing_destinations_policy: ignore` prevented corrections across redeploys. | **Changed `instance` → `subaccount` and `ignore` → `update` in mta.yaml destination-content** |

> ⚠️ **Critical rule**: In `mta.yaml` destination-content, always use `subaccount:` (not `instance:`). The SAP Approuter's dynamic routing for `html5-apps-repo-rt` routes queries the Destination Service at the **subaccount** level. Instance-level destinations are invisible to it.
> Also use `existing_destinations_policy: update` so destinations are refreshed on every deployment.

**Files changed in namespace rename fix**:
- `app/ui/webapp/manifest.json` — sap.app.id, sap.cloud.service, bundleName, viewPath
- `app/ui/webapp/index.html` — resourceroots, data-name
- `app/ui/webapp/Component.js` — import path, extend namespace
- `app/ui/webapp/controller/List.controller.js` — extend namespace
- `app/ui/webapp/controller/Detail.controller.js` — extend namespace
- `app/ui/webapp/controller/NotFound.controller.js` — extend namespace
- `app/ui/webapp/view/List.view.xml` — controllerName
- `app/ui/webapp/view/Detail.view.xml` — controllerName
- `app/ui/webapp/view/NotFound.view.xml` — controllerName
- `approuter/xs-app.json` — welcomeFile → `/mybtp.ui/index.html`
- `mta.yaml` — destination-content: `instance` → `subaccount`, `ignore` → `update`
- `resources/dist.zip` — rebuilt and copied after all source changes

## Figaf SaaS Connection (April 2026)

The app now supports two connection modes, controlled by env vars on `my-btp-app-srv`:

| Mode | Env vars required | How it works |
|---|---|---|
| **Destination** (BTP-hosted Figaf) | `FIGAF_USE_DESTINATION=true`, `FIGAF_DESTINATION_NAME=figaf-api` | Uses SAP Cloud SDK with the named destination |
| **Login** (SaaS `app.figaf.com`) | `FIGAF_BASE_URL=https://app.figaf.com`, `FIGAF_USERNAME=...`, `FIGAF_PASSWORD=...` | POSTs form-encoded credentials to `/auth/login`, Bearer token returned in response `authorization` header, cached 50 min |

Current `mta.yaml` settings (SaaS mode):
- `FIGAF_BASE_URL: https://app.figaf.com`
- `FIGAF_USE_DESTINATION: false`
- `FIGAF_AGENT_SYSTEM_ID: emil-cd3f5509`
- `FIGAF_USERNAME` and `FIGAF_PASSWORD` set via `cf set-env` (not in mta.yaml to avoid commit exposure)

**Important**: `cf set-env` values are overwritten by `mta.yaml` `properties` on every `cf deploy`. Always update `mta.yaml` before deploying; use `cf set-env` only for secrets (`FIGAF_USERNAME`, `FIGAF_PASSWORD`, `OPENAI_API_KEY`).

## Scenario Flow Field Mapping (April 2026)

B2B scenario flow data is **not** in `objectData.integrationObject` — it is in `objectData.additionalData` (a JSON string) → `b2BScenarioObjectData` nested object.

Correct field mapping:
| Column | Source field |
|---|---|
| Main Flow | `b2BScenarioObjectData.mainIflowTechnicalName` |
| Main Flow Runtime Status | `b2BScenarioObjectData.mainIflowRuntimeStatus` |
| Main Flow Runtime Version | `b2BScenarioObjectData.mainIflowRuntimeVersion` |
| Main Flow Design Version | `b2BScenarioObjectData.mainIflowDesignVersion` |
| Pre Flow | `b2BScenarioObjectData.preIflowTechnicalName` |
| Post Flow | `b2BScenarioObjectData.postIflowTechnicalName` |
| (and all `preIflow*`, `postIflow*` equivalents) | same nested object |
